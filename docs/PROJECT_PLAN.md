人間にわかりやすいコードとAIにわかりやすいコードの違いを検証するプロジェクト計画書

1. プロジェクト概要

本プロジェクトでは、人間にとって読みやすく保守しやすいコード設計が、生成AIやコーディングエージェントにとっても理解・変更しやすいのかを実験的に検証する。

検証は以下の二段階に分ける。

* 第1弾：コードの設計構造による違いを比較する
* 第2弾：プログラミング言語による違いを比較する

第1弾では使用言語を固定し、同じ機能を持つアプリケーションを異なる設計構造で実装する。そのうえで、同一のAIモデルに同じ変更課題を与え、実装精度、変更範囲、コード品質、探索コストなどを比較する。

第2弾では設計構造を固定し、同じ仕様のアプリケーションを複数のプログラミング言語で実装する。型システム、コンパイラ、言語の明示性、記述方法の自由度などが、AIによるコード変更へどのような影響を与えるかを比較する。

最終的には、AI時代における「AIが理解・変更しやすいコード」の特徴を整理し、Qiita記事として公開する。

⸻

2. 背景

人間向けのソフトウェア設計では、一般的に次のような要素が重視される。

* 責務が分離されている
* ファイルやクラスの役割が明確である
* 依存関係が整理されている
* 変更の影響範囲が限定されている
* ドメインの概念がコードに反映されている
* テストしやすい
* 将来的な変更や拡張に耐えられる

クリーンアーキテクチャ、レイヤードアーキテクチャ、ドメイン駆動設計などは、主に人間が長期間コードを保守することを前提としている。

一方、AIがコードを変更する場合は、人間とは異なる問題が発生する。

* 必要なファイルを検索できるか
* 複数ファイルの依存関係を正しく追跡できるか
* コンテキストへ必要な情報を収められるか
* 暗黙的な設計意図を推測できるか
* 既存コードの規則性を認識できるか
* コンパイラやテストの結果から自己修正できるか

そのため、人間にとって読みやすい設計と、AIにとって変更しやすい設計は、完全には一致しない可能性がある。

本プロジェクトでは、この違いを主観的な感想ではなく、同一条件でのコード変更実験によって確認する。

⸻

3. プロジェクトの目的

3.1 主目的

人間にとって理解しやすいコード設計と、AIにとって理解・変更しやすいコード設計の共通点と相違点を明らかにする。

3.2 副目的

* AIがコード変更時に失敗する原因を分類する
* ファイル分割や抽象化がAIの探索コストへ与える影響を調べる
* 型情報やコンパイラがAIの自己修正に与える効果を調べる
* AIによる実装を前提とした設計指針を考察する
* 再現可能な実験リポジトリと評価方法を公開する
* Qiitaで連載形式の記事として公開する

3.3 本プロジェクトで扱わないこと

* どのプログラミング言語が一般的に最も優れているか
* クリーンアーキテクチャ自体の優劣
* AIモデル全体の性能ランキング
* 実行性能や処理速度を中心とした言語比較
* 人間の開発速度そのものの比較
* 大規模商用システムでの最終的な結論

本実験で確認するのは、限定された条件におけるAIのコード理解・変更性能である。

第1弾：設計構造による違いの検証

4. 第1弾の研究課題

第1弾では、次の問いを検証する。

同じ機能を持つコードでも、ファイル構成や責務の分け方によって、AIのコード変更精度は変化するのか。

さらに、次の下位課題を設定する。

1. AIは1ファイルに集約されたコードを最も正確に変更できるのか
2. レイヤー分割されたコードでは、AIが必要なファイルを正しく発見できるのか
3. 機能単位で整理されたコードは、集約と責務分離の両方の利点を持つのか
4. 変更規模が大きくなるほど、設計構造による差は広がるのか
5. AIが失敗する主な原因は、コード生成能力ではなく探索や設計理解なのか

⸻

5. 第1弾の仮説

仮説A

局所的で単純な変更では、必要な情報が同じ場所に集まっている1ファイル構成が有利になる。

仮説B

複数の責務を横断する変更では、責務が整理されたレイヤー分割構成が有利になる。

仮説C

レイヤーを細かく分割しすぎると、AIが関連ファイルを見落とし、実装漏れが増える。

仮説D

機能単位構成は、関連コードの近接性と責務分離を両立するため、総合的に最も安定する。

仮説E

AIにとって重要なのはファイル数そのものではなく、変更に必要な情報を低コストで発見できることである。

6. 第1弾で使用する技術

使用言語

第1弾はTypeScriptで統一する。

TypeScriptを使用する理由

* 型情報を持っている
* AIの学習データが比較的多い
* ファイル分割が容易である
* 小規模なAPIを短期間で複数構築できる
* 静的型チェックを評価へ利用できる
* JavaやRustほど記述量が多くなりすぎない
* JavaScriptとの比較へ発展させやすい

技術構成

* TypeScript
* Node.js
* FastifyまたはExpress
* Vitest
* ESLint
* Prettier
* SQLiteまたはインメモリRepository
* npmまたはpnpm

データベースの設定差による影響を減らすため、初期検証ではインメモリRepositoryを使用する。必要に応じてSQLite版を追加する。

8. 比較する設計構造

構成A：モノリシック構成

すべての主要処理を1ファイルまたは極少数のファイルへまとめる。

monolithic/
├── src/
│   └── app.ts
├── tests/
│   └── app.test.ts
└── package.json

app.ts内に次の処理を持たせる。

* 型定義
* データ保存
* 業務ロジック
* APIハンドラ
* バリデーション
* エラー処理

目的は、必要な情報が集約されている状態がAIへどのような影響を与えるか確認することである。

構成B：レイヤー分割構成

クリーンアーキテクチャまたはレイヤードアーキテクチャに近い分割を行う。

layered/
└── src/
    ├── presentation/
    │   ├── controllers/
    │   ├── requests/
    │   └── responses/
    ├── application/
    │   └── usecases/
    ├── domain/
    │   ├── models/
    │   ├── services/
    │   └── repositories/
    ├── infrastructure/
    │   └── repositories/
    └── app.ts

目的は、責務が明確だが情報が複数箇所へ分散した構成を評価することである。

構成C：機能単位構成

Vertical Slice Architectureに近い形で、機能ごとに関連コードをまとめる。

feature-based/
└── src/
    ├── tasks/
    │   ├── create/
    │   ├── update/
    │   ├── complete/
    │   ├── archive/
    │   ├── task.ts
    │   ├── taskRepository.ts
    │   └── taskErrors.ts
    ├── users/
    └── app.ts

目的は、関連コードの近接性と責務分離を両立した構成を評価することである。

10. 第1弾の実験課題

課題1：単純な属性追加

タスクにcategoryを追加し、作成・更新・取得APIで扱えるようにしてください。

確認項目：

* 型への追加
* 入力値への追加
* 保存処理への追加
* レスポンスへの追加
* テストの追加
* 実装漏れの有無

課題2：局所的な業務ルール

完了済みタスクは担当者を変更できないようにしてください。

確認項目：

* 適切な責務へ実装したか
* APIハンドラへ直接ロジックを書いていないか
* エラー形式が既存仕様と一致しているか
* テストを追加したか

課題3：横断的な機能追加

タスクの操作履歴を保存し、タスクごとの履歴取得APIを追加してください。

履歴対象：

* 作成
* 更新
* 完了
* 担当者変更
* アーカイブ

確認項目：

* データモデル
* 保存処理
* 業務処理
* API
* エラー処理
* テスト

課題4：既存バグ修正

完了済みタスクが期限切れとして表示される不具合を修正してください。

確認項目：

* 原因箇所を正しく発見できたか
* 最小範囲で修正したか
* 回帰テストを追加したか
* 関係のない箇所を変更していないか

課題5：仕様変更

タスクを削除する代わりにアーカイブする仕様へ変更してください。

確認項目：

* 既存削除処理の調査
* 通常一覧からの除外
* 詳細取得の扱い
* 既存テストの更新
* 互換性の考慮

課題6：曖昧な機能追加

タスクを複製できるようにしてください。

意図的に詳細を減らし、次を観察する。

* 既存仕様を調査したか
* 勝手な仕様を大量に追加しなかったか
* 既存パターンへ合わせたか
* 不明点を適切に判断したか

第2弾：プログラミング言語による違いの検証

17. 第2弾の研究課題

第2弾では、次の問いを検証する。

同じ設計と機能を持つコードでも、プログラミング言語の違いによって、AIのコード変更精度は変化するのか。

下位課題は次のとおりとする。

1. 静的型付き言語は動的型付き言語より変更漏れを防ぎやすいか
2. コンパイラのエラー情報はAIの自己修正を助けるか
3. 言語機能が少なく書き方が統一された言語はAIに有利か
4. コードが短い言語はコンテキスト効率で有利か
5. 厳しい型システムは初回成功率を下げても、最終成功率を高めるか
6. AIの学習データ量と言語仕様の効果を区別できるか

⸻

18. 第2弾の仮説

仮説A

Pythonは初回実装が速く、変更行数も少ないが、実行時まで発見されない不具合が増える。

仮説B

TypeScriptは記述量と型安全性のバランスがよく、総合成功率が高くなる。

仮説C

Goは書き方が統一され、依存関係が明示されやすいため、既存コードとの一貫性が高くなる。

仮説D

Rustは初回のコンパイルエラー数が多くなるが、修正後の隠しテスト成功率が高くなる。

仮説E

AIにとって有利な言語とは、最初から正しいコードを生成しやすい言語ではなく、間違いを検出して修正しやすい言語である。

19. 比較対象言語

主要比較は以下の4言語とする。

* Python
* TypeScript
* Go
* Rust

Python

動的型付け、短い記述量、豊富な学習データを持つ言語として扱う。

TypeScript

構造的型付け、Web開発での普及、記述量と型情報のバランスを持つ言語として扱う。

Go

単純な言語仕様、明示的なエラー処理、統一されたフォーマットを持つ言語として扱う。

Rust

強力な型システム、所有権、詳細なコンパイラ診断を持つ言語として扱う。

実験量が大きすぎる場合は、最初にTypeScript、Go、Rustの3言語で実施し、Pythonを追加検証とする。

4言語 × 10課題 × 3回 = 120実験

最大構成：

4言語 × 10課題 × 5回 = 200実験

25. 追加の小規模実験

本編とは別に、原因を分離する補助実験を行う。

型注釈あり・なし

Pythonで同一コードを用意する。

* 型注釈なし
* 型注釈あり

これにより、言語そのものではなく型情報の影響を確認する。

JavaScript・TypeScript比較

同じフレームワークと似た文法で比較し、静的型情報の効果を確認する。

コンパイラ利用あり・なし

同じ課題を次の2条件で実施する。

* AIがコードを一度だけ生成する
* AIがコンパイラとテストを実行しながら修正できる

これにより、言語理解のしやすさと自己修正のしやすさを分離する。

READMEあり・なし

構成説明をREADMEへ記載した場合と、コードだけを与えた場合を比較する。

これは第3弾候補として独立させてもよい。

想定される結論

43. 第1弾で予想される結果

* 小さな変更ではモノリシック構成が強い
* 横断的変更ではレイヤー分割構成が強い
* 総合的には機能単位構成が安定する
* ファイル数より関連コードの近さが重要になる
* 過剰な抽象化はAIの探索ミスを増やす
* 一貫した命名と規則性がAIの変更精度を高める
* テストが設計意図を補う重要な情報になる

44. 第2弾で予想される結果

* Pythonは初回生成と記述量で有利
* TypeScriptは生成と検証のバランスがよい
* Goはコードの一貫性で有利
* Rustは初回失敗が多いが最終品質で有利
* 静的型は変更漏れの検出に効果がある
* 詳細なコンパイラエラーはAIの自己修正を助ける
* AI向けの言語評価では、初回成功率と最終成功率を分ける必要がある

これらは実験前の仮説であり、結果に合わせて変更してはならない。予想と異なる結果もそのまま掲載する。

リスクと対策

45. モデル更新の影響

リスク

実験途中でAIモデルが更新されると、前半と後半の結果を単純比較できない。

対策

* 短期間で同一フェーズを実施する
* モデル名とバージョンを記録する
* 実施日時を保存する
* 更新が発生した場合は別グループとして扱う

46. 実験量が多すぎる

リスク

全条件を実施すると、コストと確認作業が大きくなる。

対策

* 最初は各条件3回にする
* 差が大きい課題だけ追加実施する
* ログ収集と採点を自動化する
* 補助実験は本編後に実施する

47. 課題が簡単すぎる

リスク

すべての構成や言語で成功し、差が出ない。

対策

* 局所変更と横断変更を混ぜる
* 隠しテストを充実させる
* 状態遷移やエラー処理を含める
* 試験実行で難易度を確認する

48. 課題が難しすぎる

リスク

すべて失敗し、比較できない。

対策

* 難易度の異なる課題を用意する
* 1課題に含める要件を制限する
* 基本機能の実装例を既存コードに含める
* 全失敗した課題は参考結果として扱う

49. 主観評価の偏り

リスク

自分の好きな設計や言語へ高い点を付けてしまう。

対策

* 採点表を実験前に固定する
* 機能評価を中心にする
* 隠しテストを使用する
* 可能なら別評価者を用意する
* 評価対象の条件を伏せる
